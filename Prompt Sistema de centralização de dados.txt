## REGRA DE MULTI-UNIDADE (MUITO IMPORTANTE)

O Admin Master pode visualizar e gerenciar até 2 unidades do Colégio Objetivo.
Cada unidade possui dados completamente isolados — uma unidade jamais deve
enxergar, interferir ou se comunicar com os dados da outra.

O Admin Master pode conceder permissão a um usuário com perfil "diretor"
(sem acesso master) para visualizar as informações de mais de uma unidade,
assim como ele próprio faz.

O perfil "atendimento" NUNCA pode ter acesso a mais de uma unidade.
Essa restrição deve ser aplicada tanto no banco (RLS) quanto no frontend
(esconder a opção na UI para perfil atendimento).

Para suportar multi-unidade, a relação entre usuários e unidades deve ser
feita por uma tabela de junção (user_units), não por um único campo unit_id
na tabela profiles. O campo unit_id em profiles pode ser mantido como
unidade principal para referência, mas o controle de acesso real deve usar
a tabela user_units.

## BANCO DE DADOS — Crie todas estas tabelas no Supabase:

### Tabela: units
```sql
create table units (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  created_at timestamptz default now()
);
```
Popule com as unidades: "Objetivo Portal" e "Objetivo Baixada".

### Tabela: profiles
```sql
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  full_name text not null,
  email text not null,
  role text not null check (role in ('admin', 'diretor', 'atendimento')),
  unit_id uuid references units(id),
  is_active boolean default true,
  created_at timestamptz default now()
);
```
Constraint de domínio de e-mail:
```sql
alter table profiles add constraint valid_email_domain
  check (email ~* '^[^@]+@(objetivoportal|objetivobaixada)\.[a-z]+$');
```

### Tabela: user_units (controle de acesso multi-unidade)
```sql
create table user_units (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id) on delete cascade not null,
  unit_id uuid references units(id) on delete cascade not null,
  granted_by uuid references profiles(id),
  granted_at timestamptz default now(),
  unique(user_id, unit_id)
);
```
REGRA DE NEGÓCIO:
- Perfil "atendimento": máximo 1 registro nesta tabela (enforce via trigger).
- Perfil "diretor" sem permissão especial: 1 unidade.
- Perfil "diretor" com permissão do admin: até 2 unidades.
- Perfil "admin": acesso automático a todas as unidades via role = 'admin' no RLS.

### Tabela: invite_tokens
```sql
create table invite_tokens (
  id uuid primary key default gen_random_uuid(),
  token text unique not null default encode(gen_random_bytes(32), 'hex'),
  email text not null,
  full_name text not null,
  role text not null check (role in ('diretor', 'atendimento')),
  unit_id uuid references units(id),
  expires_at timestamptz not null default (now() + interval '1 hour'),
  used_at timestamptz,
  status text not null default 'pending' check (status in ('pending','used','expired')),
  created_by uuid references profiles(id),
  created_at timestamptz default now()
);
```

### Tabela: students
ATENÇÃO: Esta tabela foi expandida para conter todos os dados do formulário
"Novo Registro de Caso" preenchido pelo setor de atendimento.

```sql
create table students (
  id uuid primary key default gen_random_uuid(),

  -- Identificação do aluno
  full_name text not null,
  ra text not null,
  unit_id uuid references units(id) not null,

  -- Nível de ensino (enum restrito)
  education_level text not null check (
    education_level in (
      'educacao_infantil',
      'ensino_fundamental_1',
      'ensino_fundamental_2',
      'ensino_medio'
    )
  ),

  -- Série do aluno (texto livre, ex: "3º ano", "1ª série")
  serie text not null,

  -- Tipo do caso
  status text not null check (status in ('cancelamento', 'transferencia')),

  -- Interações com coordenação
  spoke_with_coordination boolean not null default false,
  coordination_reversed boolean,
  -- coordination_reversed só é preenchido se spoke_with_coordination = true.
  -- Se spoke_with_coordination = false, coordination_reversed deve ser NULL.
  -- Se spoke_with_coordination = true e coordination_reversed = false,
  -- o campo coordination_no_reversal_reason deve ser preenchido.
  coordination_no_reversal_reason text,

  -- Interação com a direção
  spoke_with_direction boolean not null default false,

  -- Controle de registro
  created_by uuid references profiles(id),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

REGRAS DE VALIDAÇÃO DO FORMULÁRIO (aplicar no frontend E via check no banco):

1. Se spoke_with_coordination = false:
   - coordination_reversed deve ser NULL
   - coordination_no_reversal_reason deve ser NULL

2. Se spoke_with_coordination = true E coordination_reversed = true:
   - coordination_no_reversal_reason deve ser NULL

3. Se spoke_with_coordination = true E coordination_reversed = false:
   - coordination_no_reversal_reason é OBRIGATÓRIO (texto explicando o motivo)

Adicionar constraints no banco:
```sql
alter table students add constraint coordination_logic check (
  (spoke_with_coordination = false
    and coordination_reversed is null
    and coordination_no_reversal_reason is null)
  or
  (spoke_with_coordination = true
    and coordination_reversed = true
    and coordination_no_reversal_reason is null)
  or
  (spoke_with_coordination = true
    and coordination_reversed = false
    and coordination_no_reversal_reason is not null
    and trim(coordination_no_reversal_reason) <> '')
);
```

## ROW LEVEL SECURITY

Ative RLS em TODAS as tabelas. Nenhuma tabela deve ter RLS desativado.

### profiles
```sql
alter table profiles enable row level security;

create policy "profiles_select" on profiles for select
  using (
    auth.uid() = id
    or exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
  );

create policy "profiles_insert" on profiles for insert
  with check (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
  );
```

### user_units
```sql
alter table user_units enable row level security;

create policy "user_units_select" on user_units for select
  using (
    user_id = auth.uid()
    or exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
  );

create policy "user_units_insert" on user_units for insert
  with check (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
  );

create policy "user_units_delete" on user_units for delete
  using (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
  );
```

### students
```sql
alter table students enable row level security;

create policy "students_select" on students for select
  using (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
    or exists (
      select 1 from user_units uu
      where uu.user_id = auth.uid()
      and uu.unit_id = students.unit_id
    )
  );

create policy "students_insert" on students for insert
  with check (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
    or exists (
      select 1 from profiles p
      join user_units uu on uu.user_id = p.id
      where p.id = auth.uid()
      and p.role = 'atendimento'
      and uu.unit_id = unit_id
    )
  );

create policy "students_update" on students for update
  using (
    exists (
      select 1 from profiles p where p.id = auth.uid() and p.role = 'admin'
    )
    or exists (
      select 1 from profiles p
      join user_units uu on uu.user_id = p.id
      where p.id = auth.uid()
      and p.role = 'atendimento'
      and uu.unit_id = students.unit_id
    )
  );
```

## FLUXO DE CONVITE (Invite-Only, sem registro público)

O sistema NÃO deve ter página de cadastro pública. O fluxo é:

1. Admin Master acessa o painel de usuários e clica em "Adicionar Usuário".

2. Admin preenche:
   - Nome completo
   - E-mail (validado: apenas @objetivoportal.com.br ou @objetivobaixada.com.br)
   - Perfil: diretor ou atendimento
   - Unidade principal
   - Se perfil for "diretor": opção de selecionar segunda unidade (opcional)
   - Se perfil for "atendimento": o admin master só poderá selecionar 1 unidade. "Atendentes só podem acessar uma unidade"

3. O sistema salva na tabela invite_tokens com expires_at = now() + 1 hora.

4. O sistema envia e-mail via Supabase com link:
   https://[dominio]/convite/[token]

5. O usuário abre o link. O sistema verifica:
   - Token existe na tabela?
   - status = 'pending'?
   - expires_at > now()?
   Se qualquer verificação falhar: tela de erro "Link inválido ou expirado.
   Solicite um novo convite ao administrador."

6. Se válido: tela com nome e e-mail pré-preenchidos (somente leitura) e
   campo para criar senha com requisitos:
   - Mínimo 8 caracteres
   - Pelo menos 1 letra maiúscula
   - Pelo menos 1 número
   - Pelo menos 1 caractere especial
   Mostrar indicador visual de força da senha em tempo real.

7. Ao confirmar:
   - Cria usuário no Supabase Auth
   - Cria registro em profiles
   - Cria registros em user_units
   - Marca token: used_at = now(), status = 'used'
   - Registra em audit_logs

8. Redireciona para /login com mensagem de sucesso.

## GESTÃO DE ACESSO MULTI-UNIDADE PELO ADMIN

Na tela /usuarios, o Admin deve ter seção "Permissões de Unidade" para
cada usuário diretor:

- Listar unidades às quais o diretor já tem acesso
- Botão "Conceder acesso a outra unidade" (disponível se tiver menos de 2)
- Botão "Revogar acesso" (desde que reste pelo menos 1 unidade)
- Para perfil atendimento: seção desabilitada com mensagem explicativa

Ao conceder nova unidade:
- Verificar se o usuário é diretor (não atendimento)
- Verificar se ele já não tem 2 unidades
- Inserir em user_units
- Registrar em audit_logs: "Acesso à unidade X concedido ao usuário Y"

## FORMULÁRIO: NOVO REGISTRO DE CASO (/alunos/novo)

Esta é a tela principal usada pelo setor de atendimento para registrar
casos de cancelamento de matrícula ou transferência. Acesso: apenas
perfis "atendimento" e "admin".

O formulário deve ser organizado em seções visuais claras com separadores,
títulos de seção e ícones. Todos os campos obrigatórios devem ter asterisco
vermelho (*) e validação em tempo real.

### SEÇÃO 1 — Identificação do Aluno

Campo 1: Nome completo do aluno
  - Tipo: input text
  - Obrigatório: sim
  - Sanitizar com DOMPurify antes de salvar
  - Placeholder: "Digite o nome completo do aluno"

Campo 2: RA (Registro do Aluno)
  - Tipo: input text
  - Obrigatório: sim
  - Placeholder: "Digite o RA do aluno"

Campo 3: Nível de Ensino
  - Tipo: select (dropdown)
  - Obrigatório: sim
  - Opções exatas (usar estes valores no banco e no frontend):
    * "Educação Infantil"    → valor: educacao_infantil
    * "Ensino Fundamental I" → valor: ensino_fundamental_1
    * "Ensino Fundamental II"→ valor: ensino_fundamental_2
    * "Ensino Médio"         → valor: ensino_medio
  - Placeholder: "Selecione o nível de ensino"

Campo 4: Série
  - Tipo: input text
  - Obrigatório: sim
  - Placeholder: "Ex: 3º ano, 1ª série, Maternal II..."
  - Sanitizar com DOMPurify antes de salvar

### SEÇÃO 2 — Tipo do Caso

Campo 5: Tipo do registro
  - Tipo: radio buttons ou select
  - Obrigatório: sim
  - Opções:
    * "Cancelamento de Matrícula" → valor: cancelamento
    * "Transferência"             → valor: transferencia
  - Exibir como dois cards clicáveis com ícone e descrição curta,
    destacando visualmente o selecionado

### SEÇÃO 3 — Interação com a Coordenação

Título da seção: "Atendimento pela Coordenação"

Campo 6: Já conversou com a coordenação?
  - Tipo: radio buttons
  - Obrigatório: sim
  - Opções: "Sim" / "Não"
  - Valor no banco: spoke_with_coordination (boolean)

Campo 7 (condicional — exibir SOMENTE se Campo 6 = "Sim"):
  Conseguiu reverter?
  - Tipo: radio buttons
  - Obrigatório: sim (quando visível)
  - Opções: "Sim" / "Não"
  - Valor no banco: coordination_reversed (boolean)

Campo 8 (condicional — exibir SOMENTE se Campo 7 = "Não"):
  Por qual motivo não foi possível reverter?
  - Tipo: textarea
  - Obrigatório: sim (quando visível)
  - Mínimo 20 caracteres
  - Placeholder: "Descreva o motivo pelo qual não foi possível reverter
    a situação com a coordenação..."
  - Contador de caracteres visível
  - Sanitizar com DOMPurify antes de salvar
  - Valor no banco: coordination_no_reversal_reason (text)

LÓGICA DE EXIBIÇÃO CONDICIONAL:
  - Quando Campo 6 muda para "Não": ocultar Campos 7 e 8, limpar seus valores
  - Quando Campo 7 muda para "Sim": ocultar Campo 8, limpar seu valor
  - Quando Campo 7 muda para "Não": exibir Campo 8
  - Usar animação suave de fade-in/fade-out ao mostrar/ocultar campos
    condicionais

### SEÇÃO 4 — Interação com a Direção

Título da seção: "Atendimento pela Direção"

Campo 9: Já conversou com a direção?
  - Tipo: radio buttons
  - Obrigatório: sim
  - Opções: "Sim" / "Não"
  - Valor no banco: spoke_with_direction (boolean)

### SEÇÃO 5 — Motivo do Cancelamento / Transferência

Título da seção: "Relato do Atendimento"

Campo 10: Motivo detalhado
  - Tipo: textarea grande
  - Obrigatório: sim
  - Mínimo 10 caracteres
  - Altura mínima: 180px, redimensionável verticalmente
  - Placeholder: "Descreva detalhadamente o motivo relatado pelo responsável
    ou pelo aluno para o cancelamento de matrícula ou transferência..."
  - Contador de caracteres visível
  - Sanitizar com DOMPurify antes de salvar
  - Este campo será salvo como o primeiro registro em student_reasons
    com approval_status = 'approved' automaticamente

### BOTÕES DO FORMULÁRIO

- Botão "Cancelar" (secundário, esquerda): volta para /alunos com modal
  de confirmação se houver dados preenchidos ("Tem certeza? Os dados
  preenchidos serão perdidos.")
- Botão "Salvar Registro" (primário, direita): faz submit do formulário
  com loading state ("Salvando...") e ícone de spinner

### VALIDAÇÃO COMPLETA DO FORMULÁRIO

Antes de submeter, validar:
1. Todos os campos obrigatórios preenchidos
2. Campos condicionais validados conforme sua lógica
3. Sanitizar com DOMPurify todos os campos de texto livre
4. Verificar que o e-mail do usuário logado pertence à unidade correta (RLS)

Após salvar com sucesso:
- Redirecionar para /alunos/:id (tela de detalhes do aluno recém-criado)
- Exibir toast: "Registro criado com sucesso!"
- Registrar em audit_logs: ação "student_created", com detalhes do aluno

## SEGURANÇA ADICIONAL

1. SANITIZAÇÃO XSS
   Importe DOMPurify. Use em TODOS os campos de texto livre antes de salvar
   e antes de renderizar. Nunca use dangerouslySetInnerHTML sem DOMPurify.
   Exemplo:
     const safeText = DOMPurify.sanitize(inputValue);
   Campos que exigem sanitização obrigatória:
   - full_name (aluno)
   - serie
   - coordination_no_reversal_reason
   - reason_text (motivo)
   - qualquer outro campo de texto livre

2. VALIDAÇÃO DE DOMÍNIO DE E-MAIL
   Regex no frontend: /^[^@]+@(objetivoportal.com.br|objetivobaixada.com.br)\.[a-z]+$/i
   Aplicar em todos os formulários com campo de e-mail antes de submeter.

3. PROTEÇÃO DE ROTAS
   Componente PrivateRoute que verifica:
   - Usuário autenticado? → se não: redireciona para /login
   - Role correta para a rota? → se não: redireciona para /dashboard com
     toast "Você não tem permissão para acessar esta área."

4. TIMEOUT DE SESSÃO
   Listener no Supabase Auth para TOKEN_REFRESHED e SIGNED_OUT.
   Se sessão expirar: redirecionar para /login com mensagem
   "Sua sessão expirou. Faça login novamente."
   Sessão expira após 8 horas de inatividade.

5. RATE LIMITING NO CONVITE
   No componente de ativação: após 3 tentativas com senha inválida,
   bloquear formulário por 30 segundos com contador regressivo visível.

6. SEM DADOS SENSÍVEIS NO LOCALSTORAGE
   Use apenas a sessão gerenciada pelo Supabase Auth.
   Não armazene tokens ou dados de usuário manualmente no localStorage.

7. CONTENT SECURITY POLICY
   No index.html:
   <meta http-equiv="Content-Security-Policy"
     content="default-src 'self';
              script-src 'self';
              style-src 'self' 'unsafe-inline';
              connect-src 'self' https://*.supabase.co;
              img-src 'self' data:;">

8. AUDIT LOG
   Criar função logAction(action, entity, entityId, details) que insere
   em audit_logs. Chamar nas ações:
   - Login bem-sucedido
   - Tentativa de login falha (3x consecutivas)
   - Cadastro de aluno (student_created)
   - Inserção de motivo (reason_added)
   - Aprovação de motivo (reason_approved)
   - Rejeição de motivo (reason_rejected)
   - Cadastro de usuário (user_invited)
   - Concessão de acesso a unidade (unit_access_granted)
   - Revogação de acesso a unidade (unit_access_revoked)
   - Logout (user_logout)

9. ISOLAMENTO DE UNIDADE
   Toda query que busca students, student_reasons ou dados sensíveis
   DEVE filtrar pela(s) unidade(s) do usuário logado usando user_units.
   O RLS garante isso no banco — nunca confiar apenas no frontend.

10. VALIDAÇÃO DUPLA DOS CAMPOS CONDICIONAIS
    Os campos condicionais da seção "Coordenação" devem ser validados:
    - No frontend: antes de submeter o formulário
    - No banco: via constraint SQL (já definida acima)
    Nunca assumir que apenas um dos dois lados é suficiente.
